# US-B2B-06: создание накладной на поступление товара

## Реализация

Реализован seller endpoint `POST /api/v1/invoices`. `seller_id` берётся из
Bearer JWT. Перед созданием накладной сервис проверяет, что список позиций не
пуст, каждый SKU существует, принадлежит авторизованному продавцу и относится к
товару в статусе `MODERATED`.

Все позиции валидируются до записи накладной и её строк. Успешно созданная
накладная имеет контрактный статус `CREATED`. Каждая позиция ответа содержит
`accepted_quantity=null`; фактическое принятое количество заполняется отдельным
flow приёмки.

## API

Запрос:

```json
{
  "items": [
    {"sku_id": "00000000-0000-0000-0000-000000000000", "quantity": 12}
  ]
}
```

Ответы:

- `201` - накладная и её позиции, статус `CREATED`, у позиций присутствует
  `accepted_quantity`;
- `400 EMPTY_ITEMS` - пустой список позиций;
- `400 SKU_NOT_MODERATED` - SKU относится к товару не в статусе `MODERATED`;
- `403 NOT_OWNER` - SKU принадлежит другому продавцу;
- `404 NOT_FOUND` - SKU не существует.

Все ошибки имеют плоскую структуру `{"code": "...", "message": "..."}`.

## Тесты

- `test_create_invoice_with_moderated_sku_returns_201`
- `test_empty_items_returns_400`
- `test_non_moderated_sku_returns_400`
- `test_others_sku_returns_403`

## ADR

Проверку статуса и владельца SKU можно разместить в Pydantic-схеме, API-view
или service-layer. Схема хорошо проверяет форму запроса, но не должна обращаться
к БД; проверка во view привязывает бизнес-правило к одному HTTP-входу. Выбран
service-layer: он централизует правило для HTTP и будущих внутренних вызовов и
выполняет всю валидацию до начала записи. Это повышает читаемость потока и
снижает риск обхода проверки при расширении API.
