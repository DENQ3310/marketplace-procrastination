# B2C Catalog Flows

Канонические user-flow для блока "Каталог" B2C-приложения.

## B2C-4. Блок похожих товаров {#b2c-4-similar-products}

### Контекст

Покупатель открывает карточку товара и листает блок "Похожие товары" вместо
возврата в каталог. Блок должен удерживать пользователя на платформе даже
если в текущей категории мало товаров.

### Эндпоинт

`GET /api/v1/catalog/products/{id}/similar`

Параметры:

- `id` (path, required) — UUID товара, для которого ищем похожие;
- `limit` (query, optional, default `10`, `1..50`) — максимум элементов в
  ответе.

Ответ — `200 OK` со списком `CatalogProductCard` (см.
`b2c/catalog/openapi.yaml`).

### Алгоритм выборки

1. По `id` находим товар и его `category_id`. Если товар не найден —
   `404 NOT_FOUND`.
2. Собираем категорию и все её подкатегории (`category descendants`) и
   выбираем из них случайную выборку видимых товаров (статус `MODERATED`,
   не удалён, есть SKU с `active_quantity > 0`), исключая сам товар.
3. **Fallback на родительскую категорию**: если из шага 2 набралось меньше
   `limit` товаров, и у категории есть `parent_id`, дополняем выборку
   товарами из родительской категории и её подкатегорий (тоже случайно,
   исключая уже выбранные и сам товар), пока не наберём `limit` или не
   останется кандидатов.
4. Полученный набор обогащается данными категории, рейтингом/отзывами и
   изображениями и возвращается как `CatalogProductCard[]`.

### Edge cases

- **Похожих нет совсем** (категория и родительская категория пусты) →
  `200 OK` с пустым списком `[]`. Фронт скрывает блок "Похожие товары".
- **Товар не найден** → `404 NOT_FOUND` с телом
  `{"code": "NOT_FOUND", "message": "..."}`.
- Текущий товар никогда не попадает в результат.
- Возвращается не более `limit` (по умолчанию до 8 на фронте) товаров.

### Сценарии (тесты)

- `similar_returns_up_to_8_from_same_category` — happy path: в категории
  >= 8 товаров, ответ содержит до 8 элементов из той же категории, текущий
  товар отсутствует в результате.
- `empty_category_returns_200_empty_list` — в категории кроме текущего
  товара никого нет и нет родительской категории → `200 []`.
- `unknown_product_returns_404` — запрос по несуществующему `id` →
  `404 NOT_FOUND`.

### Примечания

- Запрос алгоритма выборки похожих товаров не проксируется в B2B — это
  чисто B2C-функциональность поверх каталога, синхронизированного из B2B.

## B2C-5. Навигация по категориям {#b2c-5-category-nav}

### Контекст

Покупатель ориентируется в магазине через боковое меню (дерево категорий)
и хлебные крошки на странице категории/товара. Без навигации каталог
выглядит как плоский список, и покупатель не понимает, в какой части
магазина он находится.

### Структура дерева категорий

Хранение — adjacency list (`Category.parent_id`). Дерево строится из
плоского списка категорий обходом от корневых узлов (`parent_id IS NULL`)
вглубь. Каждый узел дерева содержит:

- `id`, `name`, `parent_id`;
- `level` — глубина узла, считая от `0` у корня;
- `path` — массив названий категорий от корня до текущего узла включительно;
- `children` — вложенные узлы (рекурсивно), `[]` для листа.

Дерево кэшируется (см. `cache/categories_tree.json`) и инвалидируется при
изменении категорий.

### Эндпоинты

`GET /api/v1/catalog/categories/tree`

Возвращает массив корневых `CategoryTreeNode` с вложенными `children`.

`GET /api/v1/catalog/categories/{category_id}`

Параметры:

- `category_id` (path, required) — UUID категории;
- `include_product_count` (query, optional, default `false`) — добавить в
  ответ количество товаров в категории и её подкатегориях.

Возвращает `CategoryInfoResponse` с информацией о категории, включая
ссылку на родителя (`parent`), если он есть.

`GET /api/v1/breadcrumbs`

Принимает **ровно один** из параметров:

- `category_id` (query) — UUID категории;
- `product_id` (query) — UUID товара (категория резолвится через товар).

Формат ответа — `BreadcrumbResponse`:

- `data` — массив `BreadcrumbItem` (цепочка от корня до текущей категории),
  каждый элемент содержит `id`, `slug`, `name`, `url` (накопленный путь из
  слагов), `level` (начиная с `1` у корня) и `is_current` (`true` только у
  последнего элемента);
- `meta` — `resolved_via` (`CATEGORY` | `PRODUCT`), `category_id`,
  `product_id`.

### Алгоритм построения хлебных крошек

1. Если оба параметра (`category_id` и `product_id`) переданы одновременно,
   или не передан ни один — `400 INVALID_REQUEST`.
2. Если передан `product_id`, находим товар и его `category_id`. Товар или
   категория не найдены → `404 NOT_FOUND`.
3. От найденной категории идём по цепочке `parent_id` вверх до корня
   (узла без `parent_id`).
4. Если на каком-то шаге `parent_id` указывает на несуществующую категорию
   (**orphan node**, сломанная иерархия) → `422 ORPHAN_NODE`.
5. Цепочка разворачивается от корня к текущей категории и возвращается как
   `data`, с накоплением `url` из `slug` каждого узла.

### Edge cases

- **Несуществующая категория** (`category_id` или категория товара не
  найдена) → `404 NOT_FOUND` с телом
  `{"code": "NOT_FOUND", "message": "..."}`.
- **Сломанная иерархия** (ссылка на несуществующего родителя где-либо в
  цепочке) → `422 ORPHAN_NODE` с телом
  `{"code": "ORPHAN_NODE", "message": "..."}`.
- **Оба параметра одновременно или оба отсутствуют** → `400
  INVALID_REQUEST` с телом `{"code": "INVALID_REQUEST", "message": "..."}`.
- Дерево категорий с несколькими корнями возвращает по отдельному узлу для
  каждого корня (без общего псевдо-корня).

### Сценарии (тесты)

- `category_tree_returns_nested_structure` — happy path: дерево собирается
  из плоского списка категорий с корректными `level`, `path` и `children`.
- `breadcrumbs_return_path_from_root` — happy path: цепочка хлебных крошек
  идёт от корня до запрошенной категории, последний элемент помечен
  `is_current`.
- `unknown_category_returns_404` — запрос по несуществующему `category_id`
  → `404 NOT_FOUND`.
- `orphan_node_returns_422` — категория ссылается на несуществующего
  родителя → `422 ORPHAN_NODE`.
- `ambiguous_params_returns_400` — `category_id` и `product_id` переданы
  одновременно → `400 INVALID_REQUEST`.

### Примечания

- Хранение иерархии — adjacency list (`parent_id`), а не `ltree` или
  materialized path: схема проще, и orphan node обнаруживается естественным
  образом при обходе цепочки родителей (шаг 4 алгоритма выше).
